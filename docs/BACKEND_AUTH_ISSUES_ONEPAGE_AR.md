# One-Page Brief — مشكلة 500 في تسجيل الدخول

**التاريخ:** 2026-04-23  
**من:** فريق Frontend  
**إلى:** فريق Backend

## الهدف
إزالة أخطاء `500` المتقطعة في مسار تسجيل الدخول وجعل فشل المصادقة متوقعًا (`4xx`) بدل أخطاء خادم عامة.

## الأثر الحالي
- المستخدم يرى فشل تسجيل الدخول بشكل متكرر.
- الواجهة تعرض رسائل:
  - `500 Internal Server Error`
  - أو انقطاع وصول للخادم عند الشبكة/البوابة.
- هذا يؤثر مباشرة على الاعتمادية وثقة المستخدم.

## النطاق المتأثر
- `POST /api/auth/login`
- `GET /api/auth/me` (مرتبط بالجلسة بعد الدخول)

## ما تم تثبيته من جهة Frontend (تم)
- توحيد عنوان الـ API في الفرونت.
- توجيه الواجهة للسيرفر المرفوع مباشرة.
- تحسين تصنيف رسائل الخطأ (4xx / 5xx / network).

> بالتالي: أي `500` حاليًا مرجّح أنه صادر من backend / infra.

---

## المطلوب من فريق Backend (بالأولوية)

### P0 — تشخيص وإيقاف 500
1. إضافة **Structured Logs** حول `AuthService.login`:
   - قبل/بعد جلب المستخدم من DB
   - قبل/بعد `bcrypt.compare`
   - قبل/بعد `jwt.sign`
   - قبل/بعد تحديث `lastLogin`
2. إضافة `requestId` في كل استجابة خطأ وربطه بالـ logs.
3. في `ExceptionFilter`: لا تعيد رسالة عامة فقط؛ أرفق `code` و`requestId`.

### P1 — ثبات الإعدادات
1. تأكيد ثبات `JWT_SECRET` في الإنتاج (غير فارغ، غير متغير بين deploys).
2. مراجعة:
   - `JWT_EXPIRE`
   - `JWT_COOKIE_NAME`
   - `JWT_COOKIE_SAME_SITE`
   - `JWT_COOKIE_SECURE`
   - `AUTH_RETURN_TOKEN_IN_BODY`
3. مراجعة `CORS_ORIGIN` ليتوافق مع أصل الفرونت الحقيقي.

### P1 — مرونة الخدمات التابعة
1. التعامل الصريح مع أخطاء DB/Redis/Upstream.
2. إرجاع أكواد مناسبة:
   - `401/403` لمشاكل الاعتماد
   - `429` للـ throttling
   - `503` للخدمات غير المتاحة
   - تجنب `500` إلا للحالات غير المتوقعة فعلًا.

### P2 — مراقبة ومنع التكرار
1. Dashboard لمعدل `5xx` على `/api/auth/login`.
2. Alert عند تجاوز عتبة (مثلاً >1% خلال 5 دقائق).
3. إضافة smoke test دوري لتسجيل الدخول.

---

## Definition of Done
- `POST /api/auth/login` لا يرجع `500` في السيناريو الطبيعي.
- حالات الفشل المتوقعة ترجع `4xx` برسائل مفهومة.
- وجود `requestId` في كل خطأ لتتبع فوري.
- استقرار لمدة 24 ساعة بدون spikes في `5xx` على auth endpoints.

## مخرجات مطلوبة من فريق Backend بعد الإصلاح
1. Root cause واضح (ما الذي كان يسبب 500؟).
2. PR/commit مع التعديلات.
3. لقطات metrics قبل/بعد الإصلاح.
4. نتيجة اختبار حمل خفيف لمسار تسجيل الدخول.
